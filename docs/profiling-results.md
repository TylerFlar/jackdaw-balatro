# Training Pipeline Profiling Results

Generated by `scripts/profile_training.py`. All measurements on CPU
(Intel desktop, single-threaded, no CUDA).

## Summary

| Config | SPS (before) | SPS (after) | Speedup |
|--------|-------------|-------------|---------|
| Tiny (embed=32, 1 layer, 2 envs) | 741 | **1,075** | **1.45x** |
| Small (embed=64, 2 layers, 4 envs) | 785 | **1,122** | **1.43x** |

Key optimizations applied:
- Vectorized card log-prob/entropy in `evaluate_actions` (batch tensor ops)
- Grouped entity log-probs by action type (few batched calls instead of B)
- Batched type sampling in `sample_action` (single `Categorical` for all B)
- Numpy-first collation pattern (single `torch.from_numpy` per array)
- Module-level import fix in `encode_consumable`

## Bottleneck Analysis

### Current bottlenecks (post-optimization)

1. **Policy forward/sample** (72-76% of rollout). The transformer forward pass
   and autoregressive entity/card sampling dominate. This is inherent to CPU
   inference — GPU would reduce this by 3-10x.

2. **Env step** (16-21% of rollout). At 120us per env per step, this is the
   combined cost of action conversion, engine step, observation encoding,
   and action mask computation. The engine itself is only ~25us; observation
   encoding (28us) is the main env-layer cost.

3. **evaluate_actions** (48-63% of PPO update). Still the largest PPO cost
   despite 1.8-3.4x optimization. The per-batch-item entity loop remains
   sequential — further vectorization would require padding all entity masks
   to the same action type, which is complex.

4. **backward** (27-34% of PPO update). Proportional to forward pass
   complexity; nothing actionable without model architecture changes.

### What's NOT a bottleneck

- **GAE computation**: <4ms, <1% — negligible
- **Collation**: 6-7% of rollout — acceptable after numpy-first optimization
- **Reward computation**: <2us — negligible
- **Action conversion**: <2us — negligible
- **Shop splits**: <1us — negligible

## Remaining Optimization Opportunities

| Priority | Optimization | Expected Impact | Complexity |
|----------|-------------|----------------|------------|
| 1 | GPU training (move policy to CUDA) | 3-10x on forward/backward | Config change |
| 2 | `SubprocVectorEnv` (parallel env stepping) | ~2x on env_step at 8+ envs | Medium |
| 3 | Pad entity masks for batched entity log-probs | ~30% on evaluate_actions | High |
| 4 | Cache unchanged global_context between steps | ~50% on obs_encode | Medium |
